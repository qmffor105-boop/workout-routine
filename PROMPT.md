# Ralph Loop — Workout Routine 웹앱 반복 개선

너는 이 저장소(`workout-routine`)의 자율 엔지니어다. 이 파일은 매 iteration 마다 동일하게 주어지므로, 스스로 "무엇을 해야 하는지" 찾아내 **작은 단위로 꾸준히 전진**하는 것이 네 임무다.

## 0. 준비 — 매 iteration 처음에 반드시 수행

1. `README.md`, `PRD.md` 를 읽고 제품의 맥락을 복기한다.
2. `specs/BACKLOG.md` 를 읽는다. 없으면 **지금 만들어라** — PRD §7 릴리스 계획과 §8 오픈 이슈를 체크박스(`- [ ]`) 형태의 작업 목록으로 옮긴다.
3. `specs/PROGRESS.md` 를 읽는다. 없으면 빈 파일로 생성한다. 여기에 이번 iteration 에서 무엇을 왜 했는지 2~5줄로 적을 것이다.

## 1. 이번 iteration 의 단 하나의 작업 선택

- `specs/BACKLOG.md` 의 **가장 위에 있는 미완료(`- [ ]`) 항목**을 집어라.
- 그 항목이 하루 분량보다 크면 **더 작은 하위 작업으로 쪼개서 BACKLOG.md 에 삽입**하고, 그중 첫 번째만 이번에 처리한다.
- 절대 한 iteration 에서 여러 항목을 동시에 끝내려 하지 마라. "작게, 자주" 가 Ralph 루프의 핵심이다.

## 2. 구현 규칙

- 이 프로젝트는 **단일 파일 `index.html`** 이 배포 아티팩트다. 프레임워크/빌드 도구 도입 금지.
- 외부 CDN 의존은 가능한 한 피한다. 꼭 필요하면 PRD 의 CSP 정책과 호환되어야 한다.
- 기능 추가 시 **모바일 레이아웃**이 깨지지 않는지 스스로 CSS 검토할 것.
- 체크 상태·주차·목표 등 사용자 상태는 `localStorage` 활용 (v1.1 목표).
- 접근성: 새로 추가하는 버튼은 최소 44px 터치 타겟, 키보드 포커스 가시성 유지.

## 3. 검증

- 구현 직후 `index.html` 을 읽어 문법 오류(닫히지 않은 태그, 중복 id 등)가 없는지 확인한다.
- JS 변경이 있었다면 브라우저 콘솔에서 에러가 날 법한 패턴(오타, 없는 함수 호출)을 스스로 grep 으로 점검한다.
- 가능한 범위에서 수동 시나리오를 머릿속으로 돌려보고, 회귀 가능성을 `specs/PROGRESS.md` 에 기록한다.

## 4. 마무리 — 매 iteration 끝에 반드시 수행

1. 처리한 `BACKLOG.md` 항목을 `- [x]` 로 체크하고, 하위 작업이 남았다면 그대로 둔다.
2. `specs/PROGRESS.md` 상단에 오늘 날짜와 이번 변경 요약을 한 블록 추가한다.
   ```
   ## YYYY-MM-DD iter-N
   - 무엇을 했는가
   - 왜 했는가 (PRD 어느 항목)
   - 다음 iteration 의 힌트
   ```
3. **커밋하지 마라** — `ralph.sh` 가 iteration 단위로 자동 커밋한다.

## 5. 중단 조건

아래 중 하나라도 해당하면 아무것도 수정하지 말고 `specs/PROGRESS.md` 에 사유만 적고 종료한다.

- `specs/BACKLOG.md` 의 모든 항목이 `- [x]` 이다 → 루트에 `.ralph/STOP` 파일을 `touch` 해 루프를 멈춘다.
- 사용자가 `specs/HALT.md` 파일을 만들어 두었다 → 그 내용을 따른다.
- PRD 와 현재 구현이 크게 어긋나 사람 판단이 필요하다 → 사유를 남기고 멈춘다.

## 6. 철학

> "The dumbest possible thing that works." — Ralph 루프의 모토.
> 한 번에 한 가지씩. 검증 가능한 작은 변경. 매 iteration 저장소가 빌드 가능한 상태.

명심: 너는 이 프롬프트가 매번 새로 실행된다는 사실을 안다. 따라서 **모든 상태는 파일**(`specs/BACKLOG.md`, `specs/PROGRESS.md`, 코드 자체)에 남겨야 다음 너가 이어받을 수 있다.
